# 에이전트 도구 명세서 (Tools Specification)

## 1. 개요
도구(Tools)는 에이전트가 세상과 상호작용하고 데이터를 처리하기 위한 구체적인 수단(Capabilities)입니다. 각 도구는 다른 도구에 의존하지 않고 오직 **상태(State)** 객체에만 의존하여 입력값을 얻고 결과를 기록해야 합니다.

## 2. 도구 설계 표준
- **비동기 처리**: 파일 I/O나 파싱(향후 LLM API 통신 등)이 포함되므로 모든 도구 함수는 `async`로 구현되어야 합니다.
- **상태 주입**: 모든 도구의 첫 번째 인자는 현재 작업 중인 `State` 객체여야 합니다.
- **로깅 의무**: 도구의 시작, 주요 분기점, 종료, 에러 발생 시 반드시 `state.addLog()`를 통해 이력을 남겨야 합니다.

## 3. 개별 도구 명세

### 3.1. 텍스트 추출 도구 (예: `pdfTool`)
- **입력**: `State` 객체 (내부의 `pdfPath` 활용)
- **수행 동작**: 지정된 경로의 PDF 파일을 읽어 페이지 단위로 텍스트를 추출합니다.
- **상태 변경**: 전체 페이지 수(`totalPages`)를 State에 갱신합니다.
- **출력**: 추출된 텍스트 배열을 반환합니다.

### 3.2. 데이터 분석 및 파싱 도구 (예: `parseTool`)
- **입력**: `State` 객체, 텍스트 추출 도구에서 반환된 텍스트 배열(`pages`)
- **수행 동작**: 텍스트 배열을 순회하며 정해진 패턴(또는 LLM 추론)을 통해 요구사항(ID, 명칭, 내용 등)을 객체화합니다. 페이지 루프 진입 시 `state.currentPage`를 갱신합니다.
- **상태 변경**: 객체화된 요구사항을 `state.addRequirement(req)`를 호출하여 State에 누적시킵니다. `state.includeAll` 옵션을 참조하여 파싱 범위를 결정해야 합니다.
- **출력**: 없음(void). State가 단일 진실 공급원이므로 결과는 `state.getRequirementsList()`를 통해 조회합니다. 별도의 반환 배열을 두지 않습니다.

### 3.3. 세부내용 정제 도구 (예: `refineTool`)
- **입력**: `State` 객체 (내부의 파싱 완료된 요구사항 리스트 활용)
- **수행 동작**: 추출된 요구사항 객체의 `detail`(세부내용)을 검사합니다. 여러 행으로 구성되어 있고 특정 특수문자(예: `○`, `-`, `·` 등)로 항목이 나열된 경우, 해당 특수문자를 기준으로 항목을 분리합니다. 분리된 각 항목을 별도의 요구사항 로우(Row)로 분리하며, 기존 `요구사항 정의 ID`에 `-001`, `-002` 등의 일련번호(서브 ID)를 부여합니다.
  - **"상위 항목" 판별 규칙**: 각 라인의 (공백을 제외한) 첫 비공백 문자만을 검사하여 그것이 분리 대상 특수문자(`○`, `-`, `·` 등)일 때 해당 라인을 분리 후보로 본다. 같은 `detail` 블록 내에서 등장하는 특수문자 중 **들여쓰기 깊이가 가장 얕은(=가장 왼쪽 컬럼에 위치한) 문자 종류만을 상위 항목으로 간주**한다. 그보다 더 들여쓰여진 라인은 직전 상위 항목의 본문에 그대로 귀속시킨다.
  - **라인 내 분리 정책**:
    - 한 라인에 여러 특수문자가 있더라도 라인의 **첫 번째 특수문자**만 분리 기준으로 사용한다. 그 뒤의 문자는 본문 텍스트로 취급한다.
    - **분리 대상 범위**: 요구사항 본문 중 **세부내용(`detail`) 영역만** 분리 대상으로 본다. **정의(`definition`) 영역은 저장에서 제외**(파싱 단계에서 폐기)된다.
      - 정의는 단일 행 형식(`정의 [내용]`)을 전제로 한다. 정의 마커가 등장한 **그 라인 한 줄만** `요구사항정의명`으로 하고, 이후 라인은 세부내용 자동 진입 대상이 된다(첫 bullet 라인을 만나면 `detail` 모드로 들어감).
      - 이로써 PDF에서 명시적 `세부내용` 마커가 없거나 bullet 항목보다 뒤에 오는 경우에도 첫 번째 bullet 항목이 detail에 정확히 포함된다.
    - **문맥 기반 분리 조건 (LLM 위임)**: 인덴트·bullet 기반 분리로 후보(candidate) 항목들을 도출한 뒤, **첫행의 첫번째 특수문자(=상위 bullet)** 로 시작하는 항목들을 정의ID 부여 단위로 본다. 각 항목의 그룹 귀속은 LLM에 위임한다.
      - **첫 문맥(preamble) 보존**: 첫 번째 상위 항목 bullet이 등장하기 이전의 비-bullet 라인(서두/도입 문맥)은 별도 분리 후보로 만들지 않고 **첫 번째 분리 항목의 본문에 합쳐서** 보존한다.
      - **그룹화 규칙**:
        - **첫 번째 항목**(기준 문맥)에는 항상 정의ID `-001`을 부여한다.
        - **후속 각 항목**에 대해 LLM이 첫번째 문맥과의 일치 여부를 판단한다.
          - 일치(`true`) → 동일 정의ID `-001`로 **병합**한다. 본문(`detail`)은 줄바꿈으로 이어붙여 단일 서브 요구사항으로 저장한다.
          - 불일치(`false`) → 새 정의ID를 **순차 추가**한다 (`-002`, `-003`, ...). 각 불일치 항목은 자기 자신만의 서브 요구사항으로 저장된다.
      - **분리 모드(splitMode)**: 구현체는 다음 모드를 제공해야 한다. 디폴트는 `merge`.
        - `merge` (디폴트): 모든 후속 항목을 일치로 간주 → 단일 `-001`로 병합. 분류 불필요한 경우.
        - `heuristic`: 토큰 Jaccard 유사도 등 룰 기반으로 첫 문맥과의 일치 여부 판단. **외부 API 불요**.
        - `llm`: LLM(Claude)로 문맥 판단. `ANTHROPIC_API_KEY` 필요.
        - `split-all`: 모든 후속 항목을 불일치로 간주 → 각 항목을 독립 정의ID로 분리.
      - **폴백**: LLM 모드에서 API 키 부재나 호출 실패 시 `merge` 모드로 폴백한다(단일 `-001`).
    - LLM 호출은 비용 최소화를 위해 후보 단위로 묶어 배치 호출하는 것을 권장한다.
  - **원본 ID 처리 정책**: 모든 요구사항은 **디폴트로 `-001` 접미사**가 부여된다.
    - 분리가 발생하지 않은(LLM "keep" 또는 후보 미해당) 요구사항도 원본 ID 대신 `REQ-XXX-001` 형태로 저장한다.
    - 분리가 발생한(LLM "split") 요구사항은 `REQ-XXX-001`, `REQ-XXX-002`, ... 순서로 서브 ID를 부여하며 원본 ID(`REQ-XXX`)는 저장소에서 제거한다.
    - 결과적으로 저장소의 모든 요구사항 ID는 `REQ-XXX-NNN` 형태로 통일된다.
- **상태 변경**: State의 요구사항 저장소(Map)를 세분화된 요구사항 객체들로 갱신합니다.
- **출력**: 없음(void). 결과는 `state.getRequirementsList()`를 통해 조회합니다.

### 3.4. 파일 저장 도구 (예: `excelTool`)
- **입력**: `State` 객체 (내부의 `templatePath`, `outputPath`, `요구사항 리스트` 활용)
- **수행 동작**: 추출된 요구사항 배열을 지정된 엑셀 템플릿에 맞춰 서식을 유지한 채 기입하고 새로운 파일로 저장합니다.
- **상태 변경**: 결과 상태에 따른 로그 업데이트.
- **출력**: 실제로 엑셀에 기록된 행(Row)의 개수를 반환합니다.
